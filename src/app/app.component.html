What are Observables?

An observable basically can be thought of as a data source.

Various Data Sources
(User Input) Events, Http Requests, Triggered in Code, ... and so on.

Now in our Angular project, an observable basically just is an object we import from a third-part package, RxJS. The observable here is implemented in a way that it follows the observable pattern, so we have an observable and we have an observer. In between, we have a streem, a timeline

===============arrow==================>

and on this timeline, we can have multiple events emitted by the obserable or data packages you could say emitted be the observable, depending on the data source of that observable of course.

So the observable could emit data because you trigger it to do so, you can do that programmatically, it could be connected to a button and therefore whenever the button is clicked, an event in a data package is emitted automatically or as the Angular HTTP service does it, it's connected to a HTTP request. So when the response returns, the response is emitted as a data package and there are dozens of other data sources too.

So, the other part is the Observer

Observer - this actually is your code you could say. It's the subscribe function you saw earlier.

There you have three ways of handling data packages - because these are the three types of data packages you can receive and in these hooks, in these boxes you could say, you write your code gets executed.

Handle Data
Handle Error
Handle Completion
Observer
  1. You can handle the normal data;
  2. You can handle errors;
  3. You can handle the completion of the observable

So you can determine what should happen if I receive a new data package, what should happen if I receive error, what should happen when the observable eventually completes? Side note, an observable doesn't have to complete. There are observable, for example hooked up to a normal button, which never complete because how would you know when it completes? A user could click the button how ofter the user wants right.

Other observable, like the HTTP request on the other hand, will have a clear end and will complete eventually because once the response is there, what else should happen, it's done.

So this is how the observable pattern generally works and of course you use it to handle asynchronous tasks because all these data sources here, user events, triggered in your code or a HTTP request are asynchronous tasks, you don't know when they will happen and you don't know how long they will take.

Operators

Операторы позволяют трансформировать или фильтровать данные которые емитит Observable.

You might want to transform or filter data and of course, you could do that inside of you subscription or in the function you pass to your pass to your subscription, but there's a more elegant way, you can use built-in in between subscription and data from Observable. That means that the data points first of all reach operators that do something to the data and that something can be anything, there are tons of build-in operators in RxJS library and then you subscribe to the result of these operators.
